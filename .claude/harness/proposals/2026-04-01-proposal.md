### [高优先级] P1：workflow 字段在 TRACE 注释中几乎从未被填写

**问题描述：**
metrics.json 的 `by_workflow` 显示 54 条 accepted outcome 归属于 `"unknown"` 工作流，仅有 session-001 的第一条 trace（`trace_id: 2026-03-31-session-001-0000`）正确设置了 `workflow: "rule_modification"`。这意味着 proposer 无法按工作流分析失败模式，核心的"失败可追溯到具体 harness 决策"功能无法实现。

**证据 trace：**
- `2026-03-31-session-001-0000`：`"workflow": "rule_modification"`（唯一正确填写的）
- `2026-03-31-session-002-0000`：`workflow` 字段缺失
- `2026-03-31-session-004-0000`：`workflow` 字段缺失

**根因定位：** `harness_rule_ref: meta.0.2.trace_recording_rules`

当前 CLAUDE.md 第 0.2 节的 TRACE 注释示例只展示了格式，但没有在每个工作流的步骤中明确提示"此步骤的 workflow 值是什么"。Claude 在实际执行时容易省略 `workflow` 和 `harness_rule_ref` 字段。

**建议 diff（CLAUDE.md 第二层，流程 1 步骤 5）：**

```diff
- 5. **反馈结果**（含 TRACE 注释）

+ 5. **反馈结果**，在响应末尾附加：
+    <!--TRACE:{"workflow":"rule_modification","step":"step5.feedback","harness_rule_ref":"workflow.rule_modification.step5","confidence":0.9,"pre_checks":[{"check":"terminology_scan","result":"pass","findings":[]},{"check":"consistency_scan","result":"pass","findings":[]}]}-->
```

对流程 1-5 各添加对应的完整 TRACE 示例（而非只在第零层给抽象格式）。

**预期效果：** workflow 字段覆盖率从 ~3% 提升至 >80%，使下一轮 proposer 能进行有意义的按工作流分析。

**问题描述：**

**证据 trace：**

- `2026-03-31-session-001-0001`：`{"action_type": "analysis"}` — 同上

TRACE 注释格式复杂，字段多，Claude 在实际执行时会简化输出。需要在格式上降低填写门槛，在工作流步骤中提供可直接复制的完整模板。

**建议 diff（CLAUDE.md 第 5.2 节）：**

```diff
- ### 5.2 Trace 注释格式
- 每次执行关键行动后，在响应末尾附加：
- <!--TRACE:{"workflow":"<名>","step":"<步>","harness_rule_ref":"<ref>","confidence":<0-1>,"pre_checks":[]}-->

+ ### 5.2 Trace 注释模板库
+
+ **规则修改（流程1）生成计划：**
+ <!--TRACE:{"workflow":"rule_modification","step":"generate_plan","harness_rule_ref":"workflow.rule_modification.step2","confidence":0.85,"pre_checks":[{"check":"terminology_scan","result":"pass","findings":[]},{"check":"consistency_scan","result":"pass","findings":[]}]}-->
```

**预期效果：** harness_rule_ref 覆盖率从 0% 提升至 >70%，pre_checks 填写率从 0% 提升至 >60%。

### [中优先级] P3：测试数据污染——需要 is_test 标记机制

**问题描述：**
session-003 中的 49 条 trace 均为 harness 验证测试生成的虚假数据（`"test": 1`、`"test": 2` 等），占总 trace 数的 83%。这些数据拉高了接受率，掩盖了真实创作工作流的性能表现。`out-2026-03-31-0057`（`verdict: modified`，`rule_update`）是目前唯一一条来自真实创作工作流的修改反馈，但因 `workflow: null` 无法归因。

**证据：** `2026-03-31-session-003-0000` 至 `0048` 均含 `"test": N` 标识符。

**根因定位：** `harness_rule_ref: scripts.validate-harness.test_generation`

harness 脚本在验证阶段生成了大量测试 trace，但没有机制将其标记为测试数据以便在分析时过滤。

**建议 diff（aggregate-metrics.sh）：**

```diff
- recent = [o for o in outcomes if o.get('timestamp_utc', '') >= window_start
-           and o.get('scope') == 'action']
+ recent = [o for o in outcomes if o.get('timestamp_utc', '') >= window_start
+           and o.get('scope') == 'action'
+           and not o.get('is_test', False)]
```

**预期效果：** 下一轮分析将仅统计真实创作交互数据，真实接受率可精确计算。
