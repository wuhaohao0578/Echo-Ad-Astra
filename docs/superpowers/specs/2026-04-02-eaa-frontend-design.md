## 1. 执行摘要

### 1.1 项目目标

为 Echo Ad Astra TRPG 创作项目构建一个可视化伴侣工具，将人类直觉转化为结构化指令，无缝传递给 Claude Code 执行。

### 1.2 核心价值

- **降低认知负担**：可视化界面替代记忆文件路径和命令语法
- **质量保证**：实时一致性检查和术语对齐
- **工作流加速**：Harness 管理自动化，减少手动脚本调用
- **创作连续性**：灵感库保存零散想法，支持自由探索

### 1.3 关键约束

- 本地优先运行，数据不离开用户设备
- 架构必须预留 Electron 迁移路径
- AI 运算依赖 Claude Pro（API Key 多提供商配置推迟到 v2.0）
- 严格遵循 Harness 工作流语义

### 2.2 Electron 迁移路径

后端设计为独立进程，通过标准 HTTP/WebSocket 暴露 API。Electron 封装层只需：
1. 将前端打包为桌面应用
2. 在主进程启动 Node.js 后端
3. 通信层无需改动

迁移工作量估算：**2-3 周**

| 模式 | 说明 | 目标版本 |

| Electron | 原生桌面应用 | v1.1+ |

## 3. 核心概念

### 3.1 「面向 AI」翻译层

用户操作 → 可视化界面 → 结构化指令 → Claude Code

**示例流程：**

```
用户操作：
  [点击「添加新武器」按钮]
  [填写名称：「等离子步枪」]
  [拖滑块设定伤害：3D6+2]
  [填写描述：...] ↓

前端生成指令：
  {
    "type": "rule_modification",
    "action": "add_equipment",
    "workflow": "rule_modification",
    "step": "generate_plan",
    "target_path": "rules/05-装备与物品/武器.md",
    "content": {
      "name": "等离子步枪",
      "damage": "3D6+2",
      "description": "..."
    },
    "pre_checks": ["terminology_scan", "consistency_scan"]
  } ↓

Claude Code 执行：
  运行工作流 1（规则修改）
  遵循 CLAUDE.md 第三层护栏
  生成 TRACE 注释
```

### 3.2 灵感库便签

```
{
  "id": "uuid",
  "content": "星际联邦的法律体系可能借鉴了...",
  "tags": ["世界观", "政治"],
  "status": "active | classified | ai_generated",
  "confidence": 0.0-1.0,  // 自动分类置信度
  "created_at": "ISO8601",
  "last_modified": "ISO8601"
}
```

### 3.3 回传队列指令

```
{
  "id": "uuid",
  "type": "rule_modification | novel_update | harness_command",
  "action": "string",
  "payload": {},  // 具体数据结构因 type 而异
  "status": "pending | executing | success | failed",
  "affected_files": [],  // 执行后填充
  "error": null | string,
  "created_at": "ISO8601"
}
```

### 3.4 工作模式

| 模式 | 左栏 | 中栏 | 右栏 | 典型场景 |
|------|------|------|------|---------|
| **规则设计** | 文件树（rules/） | 结构化编辑器 | 回传队列 | 添加武器/修改战斗规则 |
| **小说创作** | 文件树（fiction/） | Markdown 编辑器 | 灵感库 + 回传队列 | 续写章节/术语对齐 |
| **平衡调整** | 数值表导航 | 可视化图表 | 平衡建议列表 | 调整装备价格/DPS |
| **Harness 管理** | traces/ 导航 | 仪表盘 | 操作面板 | 查看趋势/提案审批 |

## 4. 功能模块设计

### 4.1 模块 A：规则编辑器

**输入：** 用户在表单/界面中填写规则内容
**输出：** 结构化指令（类型：`rule_modification`）

**子功能：**

| 功能 | 说明 | 优先级 |
|------|------|-------|
| A.1 新增规则 | 向导式创建，自动生成文件路径 | P0 |
| A.2 编辑现有规则 | 解析 Markdown，填充结构化表单 | P0 |
| A.3 一致性检查 | 实时高亮与 `.claude/terminology.yaml` 冲突的术语 | P0 |
| A.4 术语提示 | 输入时显示可用的规范术语 | P1 |
| A.5 预览渲染 | Markdown 实时预览 | P1 |
| A.6 版本对比 | 与上一版本的差异视图 | P2 |

**数据流：**

```
用户编辑 → 前端验证 → 术语扫描 → 生成指令 → 加入回传队列
                                    ↓
                         Claude Code 执行 → 更新文件 → 返回结果
                                    ↓
                         更新 UI 状态 → 显示成功/失败
```

### 4.2 模块 B：小说创作助手

**输入：** 用户粘贴文本/使用 AI 续写
**输出：** 结构化指令（类型：`novel_update`）

**子功能：**

| 功能 | 说明 | 优先级 |
|------|------|-------|
| B.1 章节导航 | 树形视图展示所有章节 | P0 |
| B.2 AI 续写 | 调用 Claude API，风格四轴对齐 | P0 |
| B.3 术语对齐 | 自动检测非规范术语，显示替换建议 | P0 |
| B.4 风格分析 | 显示四轴匹配度（句子节奏/叙事距离/技术密度/结构） | P0 |
| B.5 灵感引用 | 从灵感库拖拽引用到编辑器 | P1 |
| B.6 世界观快查 | 侧边栏显示相关世界观条目 | P1 |
| B.7 修订历史 | 显示章节的 git 提交历史 | P2 |

**AI 续写流程（工作流 2）：**

```
读取目标文件 200 字样本 → 四轴分析 → 生成续写 → 术语对齐 → 风格匹配检查
                                     ↓
                        任一维度 < 3/5 → 重新生成（最多 3 次）
                                     ↓
                        生成新便签（标记「AI 辅助」）→ 用户可采纳或丢弃
```

### 4.3 模块 C：Harness 管理面板

**输入：** 用户点击按钮/选择选项
**输出：** Harness 脚本调用指令

**子功能：**

| 功能 | 说明 | 优先级 |
|------|------|-------|
| C.1 提案触发 | 调用 `propose.sh`，显示生成进度 | P0 |
| C.2 评分记录 | 快捷按钮调用 `score-outcome.sh` | P0 |
| C.3 Trace 历史 | 时间线视图，可过滤工作流 | P0 |
| C.4 候选管理 | 查看提案列表，一键安装候选 | P0 |
| C.5 会话统计 | 显示当前会话的行动数、通过率 | P1 |
| C.6 Trend 图表 | outcome 质量趋势（折线图） | P1 |

**与 Harness 脚本映射：**

| UI 操作 | Harness 脚本 | 参数 |
|---------|-------------|------|
| 「运行改进提案」 | `propose.sh` | `--force`（可选） |
| 「接受」 | `score-outcome.sh` | `--verdict accepted` |
| 「拒绝」 | `score-outcome.sh` | `--verdict rejected` |
| 「评分 N」 | `score-outcome.sh` | `--verdict accepted --quality overall:N` |

### 4.4 模块 D：数值平衡工具

**输入：** 用户调整数值/查看图表
**输出：** 平衡性报告和调整建议

**子功能：**

| 功能 | 说明 | 优先级 |
|------|------|-------|
| D.1 武器表 | 表格视图，支持排序/过滤 | P0 |
| D.2 DPS 计算器 | 自动计算伤害期望值 | P0 |
| D.3 价格比图表 | 每点价格伤害的对比柱状图 | P0 |
| D.4 异常检测 | 标注偏离平均值 > 30% 的装备 | P0 |
| D.5 批量调整 | 拖拽调整多项数值，实时预览影响 | P1 |

**计算公式：**

```
伤害期望值 = floor(D * (Y+1)/2) + Z  （对于 XdY+Z）
每点价格伤害 = 伤害期望值 / 价格
偏离百分比 = (当前值 - 同类平均值) / 同类平均值 * 100%
```

### 4.5 模块 E：术语表编辑器

**输入：** 用户在表格中添加/修改术语
**输出：** YAML 文件更新

**子功能：**

| 功能 | 说明 | 优先级 |
|------|------|-------|
| E.1 表格视图 | 分组显示（阵营/科技/地点/机制/角色） | P0 |
| E.2 添加术语 | 表单输入，自动验证格式 | P0 |
| E.3 影响分析 | 删除/修改前显示受影响的文件数 | P0 |
| E.4 批量替换 | 确认后生成替换计划，一次性应用到所有文件 | P0 |
| E.5 别名管理 | 添加/删除别名，自动去重 | P1 |

**影响分析流程：**

```
术语变更 → Grep 扫描所有规则/小说文件 → 显示命中文件列表
                                              ↓
                                  用户确认 → 生成替换计划 → 执行
```

### 4.6 模块 F：版本发布向导

**输入：** 用户点击「发布新版本」
**输出：** 发布流程执行

**子功能：**

| 功能 | 说明 | 优先级 |
|------|------|-------|
| F.1 发布前检查 | 验证提交状态/冲突/测试 | P0 |
| F.2 变更日志草稿 | 自动读取 traces/ 和 CHANGELOG 生成 | P0 |
| F.3 分步确认 | 每一步都需要用户确认 | P0 |
| F.4 防误操作 | 高亮显示「删除文件」等危险操作 | P0 |
| F.5 执行反馈 | 显示每一步的执行结果 | P0 |

**步骤映射（工作流 5）：**

```
步骤 1: 检查 → 显示检查结果
步骤 2: 草稿 → 自动提取 traces/ 内容
步骤 3: 确认 → 用户编辑草稿
步骤 4: 执行 → 移动 PDF / 创建标签 / 更新 README
步骤 5: 完成 → 显示发布成功
```

### 4.7 模块 G：Harness 质量仪表盘

**输入：** 用户打开仪表盘
**输出：** 趋势图表和统计卡片

**子功能：**

| 功能 | 说明 | 优先级 |
|------|------|-------|
| G.1 接受率卡片 | 显示各工作流的接受率 | P0 |
| G.2 趋势图表 | outcome 质量趋势（时间轴） | P0 |
| G.3 工作流对比 | 条形图对比各工作流表现 | P0 |
| G.4 最近错误 | 列出最近的 rejected 行动 | P1 |
| G.5 性能指标 | 平均响应时间、行动频率 | P2 |

**数据来源：**

```
.claude/harness/scores/*.json → 聚合统计 → 图表渲染
.claude/harness/traces/*.json  → 时间序列 → 趋势线
```

## 6. 数据流设计

### 6.1 回传队列执行逻辑

```
用户点击「传递给 EAA」
          ↓
队列开始执行（串行）
          ↓
┌─────────────────────────────────────┐
│ 每条指令：                           │
│ 1. 设置 status = executing           │
│ 2. 调用 Claude Code                  │
│ 3. 等待响应                          │
│ 4. 记录 affected_files / error       │
│ 5. 设置 status = success / failed    │
└─────────────────────────────────────┘
          ↓
    如果某条指令 failed
          ↓
    暂停后续执行
          ↓
    提示用户：「指令 [名称] 失败：[错误]
               [继续执行后续] [取消全部]」
          ↓
    用户选择后继续
```

### 6.2 Claude Code Bridge

前端 → 后端 → Claude Code CLI

**通信方式：**

1. **写入指令文件**（`/tmp/eaa-command.json`）
2. **触发 Claude Code**（通过 CLI hook 或文件监听）
3. **读取结果**（`/tmp/eaa-result.json`）

**前端请求示例：**

```json
POST /api/claude/execute
{
  "workflow": "rule_modification",
  "step": "generate_plan",
  "payload": {
    "type": "rule_modification",
    "action": "add_equipment",
    "target_path": "rules/05-装备与物品/武器.md",
    "content": { ... }
  },
  "pre_checks": ["terminology_scan", "consistency_scan"]
}
```

**后端响应示例：**

```json
{
  "status": "success",
  "affected_files": ["rules/05-装备与物品/武器.md"],
  "trace_id": "trace-uuid-1234",
  "confidence": 0.9,
  "pre_checks": [
    { "check": "terminology_scan", "result": "pass", "findings": [] },
    { "check": "consistency_scan", "result": "pass", "findings": [] }
  ]
}
```

### 6.3 灵感库自动分类

AI 分析便签内容 → 输出分类和置信度

**请求：**

```json
POST /api/ai/classify-note
{
  "content": "星际联邦的司法体系可能借鉴了...",
  "available_tags": ["世界观", "机制", "角色", "科技", "地点"]
}
```

**响应：**

```json
{
  "suggested_tag": "世界观",
  "confidence": 0.85,
  "alternative_tags": [
    { "tag": "政治", "confidence": 0.72 }
  ]
}
```

**UI 行为：**

- `confidence >= 0.8`：自动应用分类，不提示
- `0.5 <= confidence < 0.8`：显示建议，用户可快速确认
- `confidence < 0.5`：显示「待分类」，用户必须手动指定
